# Release Notes — v0.2.0 x402 Stable

Date: 2026-06-07  
Status: Stable checkpoint candidate  
Release type: Paywall, wallet-session, gated asset access, and demo-readiness stabilization

## Summary

This release stabilizes the Nexus RWA x402 USDC checkout and Pro wallet-session access flow.

The main outcome:

```text
A user can pay with USDC through x402, receive an active Pro session, refresh the page, and access gated asset layers without paying again until the session expires.
```

This release also improves API docs, pricing CTA routing, active Pro badge visibility, and flagship asset demo readiness.

## Why this release matters

Before this checkpoint, Nexus RWA had working asset data and dashboard layers, but the monetization/access layer was still unstable. Some gated endpoints could return 402 even when the wallet already had an active Pro session, and some frontend components expected response shapes that differed from backend live responses.

This release fixes the major post-checkout issues and makes Nexus RWA feel closer to a usable paid product.

## Stable user flow

Expected Pro flow:

1. User opens asset detail page.
2. User connects wallet.
3. User attempts to open a gated Pro layer.
4. API returns 402 if wallet has no active session.
5. Paywall launches x402 USDC checkout.
6. Payment/facilitator verification succeeds.
7. Session is persisted in Postgres.
8. Frontend retries gated request.
9. Pro layer unlocks.
10. User can refresh browser and remain Pro until `expiresAt`.

## Confirmed stable behaviors

- x402 payment flow reaches Pro status.
- Facilitator redirects are handled.
- EIP-712 token naming aligns with USDC.
- x402 payment headers are not reused incorrectly.
- Redis is disabled by default and is not a blocker.
- Wallet sessions are persisted in Postgres.
- `/v1/session?wallet=...` returns active Pro session after payment.
- Frontend retries gated fetch when wallet session is active.
- Gated asset detail `/full` sends `X-Wallet-Address`.
- Risk endpoint response shape is normalized on frontend.
- History endpoint array response is handled by frontend.
- AI insight has local fallback and should not block UI with provider failure.
- Top-bar session badge clearly shows `PRO ACTIVE` or `ENTERPRISE ACTIVE`.
- Pricing CTAs route users into real product flows.
- API docs are aligned with wallet-session access.

## Backend/session behavior

Session source of truth is Postgres.

Redis is disabled by default and should not block x402 session access.

Expected session check:

```bash
curl "https://api.nexusrwa.xyz/v1/session?wallet=0x55353bD0E4d6784d0FA7df340769D2E0375cdf7F" \
  -H "X-Wallet-Address: 0x55353bD0E4d6784d0FA7df340769D2E0375cdf7F"
```

Expected response:

```text
tier: pro
active: true
expiresAt: future timestamp
```

## Known limitations

### Dataset quality is not uniform

Not all assets are institutional-grade. The product should be described as evidence-based RWA intelligence with classification levels:

```text
institutional
analytics
research
```

Avoid implying all assets are institutional-grade.

### BUIDL is not yet institutional-grade in current scoring

BUIDL is brand-important and useful for demo, but current baseline is analytics-grade due to evidence gaps around reserve breakdown, audit/report URL, legal document URL, and proof-of-reserves.

### XAUT should not lead demos yet

Tether Gold / XAUT currently has hard blockers and should remain in research/backlog until legal/custody/reserve evidence is improved.

### AI insight depends on provider availability

Local fallback exists and should prevent 500 errors, but live AI output may vary depending on provider availability and configuration.

### History shape is currently array-based

The frontend supports the current array response. If backend later changes to a wrapped object like `{ history, limited_history }`, frontend already has a normalizer for both shapes.
